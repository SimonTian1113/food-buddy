# 🍴 食通天 FoodBuddy


[![Python 3.8+](https://img.shields.io/badge/python-3.8+-blue.svg)](https://www.python.org/downloads/)
[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](https://opensource.org/licenses/MIT)

**FoodBuddy 不是帮你找到好吃的，是帮你少踩一个雷。**

当你被小红书、抖音种草了某家店，心里却嘀咕"会不会踩雷"时——FoodBuddy 跨平台采集真实信号，交叉验证，去滤镜、识套路，给你一个尽量明确的判断。

---

## 🔥 为什么需要这个？

> **数据是平台的护城河，也是 AI 的天花板。每个 AI 都在自家的数据里答得越自信，你越该怀疑——它根本没看过隔壁的房间。**

你问豆包"上海有什么好吃的"，它给你推抖音生态里最火的内容。你问千问，它给你推高德里评分最高的店。你问问小团，它给你推大众点评必吃榜。你问点点，它给你推小红书上笔记最多的地方。

*四个AI，通常是四套答案。*

“豆千团点”们没有在骗你，它们只是在各自的信息茧房里，把"自己能看到的"当成了"全部真相"。

### 中国本地生活的数据围墙

| 厂商 | 产品 | 他们手里的数据 | 他们不会告诉你什么 |
|------|------|---------------|-------------------|
| 美团 | 问小团 | 大众点评海量商户评分、必吃榜、真实消费数据 | 不会告诉你"这家可能是刷的" |
| 字节 | 豆包 | 抖音本地生活、短视频种草数据 | 不会帮你查小红书上是不是也这么说 |
| 阿里 | 千问 | 高德本地生活、到店商户评分与位置数据 | 不会对比美团上的评分 |
| 小红书 | 点点 | 种草笔记、探店内容、用户偏好 | 不会告诉你"火"可能只是营销密度高 |

**没有谁有谁的后台数据。** FoodBuddy 希望可以用搜索引擎的摘要当桥梁（也许还有更好的办法，期待各位技术大佬支招），把原本不互通的平台评分在同一张表里对齐。

**矛盾比一致更有价值。** 小红书说"绝绝子"但 高德”10条评价中有6条差评“？这个信号，任何单个平台永远不会给你。

**如果跨平台都找不出问题，至少说明这家店问题不算太大。**

## ✨ 核心能力

### 双区域平台体系

根据城市自动切换平台权重和搜索策略：

#### 🇭🇰 港澳模式（香港 / 澳门）

| 平台 | 权重 | 定位 | 采信原则 |
|------|------|------|----------|
| OpenRice | 30% | 香港本地食客真实评价 | 本地口碑最可信，权重最高 |
| Google Maps | 25% | 全球通用评分 | 评分相对客观，参考性强 |
| 大众点评 | 20% | 内地游客视角 | 有参考性，但口味可能有偏差 |
| 小红书 | 15% | 热度参考 | 营销属性强，"火"≠"好"，谨慎采信 |
| TripAdvisor | 10% | 国际游客评价 | 游客偏差大，参考价值最低 |

#### 🇨🇳 内地模式（北京 / 上海 / 广州 / 深圳 / 成都 / 杭州 / 长沙 / 重庆 / 武汉 / 南京 / 西安）

| 平台 | 权重 | 背后厂商 | 采信原则 |
|------|------|---------|----------|
| 大众点评 | 20% | 美团 | 消费评价最全，但刷评风险存在 |
| 抖音 | 20% | 字节 | 视频种草+本地生活，营销密度高 |
| 小红书 | 20% | 小红书 | 种草内容多，"火"≠"好" |
| 高德 | 20% | 阿里 | 本地生活到店入口，地图评价+商户数据 |
| 综合 | 20% | 新闻/旅游/微博 | 第三方视角补充 |

> *四个厂商各 20%，不是因为它们一样好。*
> *是因为在这个信息不互通的世界里，谁也不比谁更值得信任。*
> *当所有声音一样大时，矛盾本身就变成了最响的那个信号。*

### 营销噪音过滤

7 层检测逻辑自动识别并过滤：

| 层级 | 检测内容 | 处理方式 |
|------|----------|----------|
| 1 | 标题级营销词（"绝绝子""yyds""全网爆火"等 28 个关键词） | 标记为重度营销 |
| 2 | Snippet 营销词快检（14 个短关键词） | 标记为轻度营销 |
| 3 | URL 域名黑名单（知乎、马蜂窝、百家号等 11 个 SEO 农场站） | 直接丢弃 |
| 4 | URL 域名白名单（Google Maps、大众点评、OpenRice 等） | 保留但检查营销信号 |
| 5 | 标题聚合页过滤（"排行榜""合集""攻略""必吃榜"） | 丢弃 |
| 6 | 可信平台保留（白名单域名即使有营销标记也保留，加 ⚠️） | 降权但保留 |
| 7 | 最终分级（重度→过滤 / 轻度→🟡标记 / 干净→无标记） | 分级输出 |

### 老饕探案组 — 6 位 AI 探员

不是普通推荐系统，是一支**老饕探案组**。组长是退休美食编辑，嘴刁、眼光毒、不信广告信证据。手下 6 位探员，各有绝活：

| 探员 | 花名 | 人设 | 职责 |
|------|------|------|------|
| 🔍 线索猎犬 | 闻锅 | 嗅觉极强的侦查员 | 跨平台采集证据，闻得出锅气也闻得出水分 |
| 🔥 拆台师 | 戳泡 | 嘴毒但证据更毒 | 专戳"看起来很厉害"的泡泡，找平台矛盾 |
| 📝 评论法医 | 验词 | 冷静挑剔的细节控 | 验评论的尸检报告，找不可伪造信号 |
| 🏠 街坊雷达 | 熟门 | 本地混久了的朋友 | 判断是本地复购盘还是游客打卡盘 |
| 🎭 场景裁剪师 | 挑局 | 会在冲动前拦朋友的人 | 判断这次去值不值，不是不能吃但不适合你 |
| ⚖️ 收口官 | 定盘 | 见过太多网红店还清醒的人 | 拍板定案：值得去 / 谨慎去 / 不建议 |

### 加权裁决算法

```
综合评分 = Σ(平台信号强度 × 平台权重) / 总权重
营销惩罚：重度营销 -0.15，中度 -0.08
裁决阈值：≥0.7 值得去 / ≥0.4 谨慎去 / <0.4 不建议
```

## 🚀 快速开始

### 安装

```bash
# 克隆仓库
git clone https://github.com/SimonTian1113/food-buddy.git
cd food-buddy

# 安装依赖（可选，用于浏览器搜索模式）
pip install playwright requests beautifulsoup4
playwright install chromium
```

### 运行

```bash
python3 scripts/orchestrator.py
```

## 📝 输出格式

每次验证输出固定 5 个模块：

1. **结论** — 值得去 / 谨慎去 / 不建议 + 踩雷风险 + 置信度
2. **为什么这样判断** — 跨平台一致性 + 主要矛盾 + 关键证据
3. **风险标签** — 网红过热 / 游客店 / 价格不透明 / 证据不足
4. **适合谁 / 不适合谁**
5. **一句话建议**

## 📝 局限性

1. **拿不到完整数据** — 反爬限制，只能拿摘要
2. **搜索摘要本身有偏差** — 算法遗漏、新店搜不到
3. **无法覆盖所有餐厅类型** — 路边摊、高端私房、连锁分店差异
4. **评分≠口味** — 判断的是"值不值得去"，不是"好不好吃"
5. **营销话术在进化** — 静态词库非常非常有可能漏检
6.  **时效性限制** — 数据是快照，出发前建议再查
